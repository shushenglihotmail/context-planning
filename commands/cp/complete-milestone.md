# /cp-complete-milestone

You are running `cp-complete-milestone`. A milestone has finished all its
phases; your job is to archive it cleanly so the project state shows the
milestone shipped and the next milestone has a clean slate.

This command is destructive in the GSD-compatible way: it COLLAPSES the
shipped phases inside `<details>` in ROADMAP.md (does NOT delete them),
APPENDS a digest to MILESTONES.md, and DELETES `.planning/MILESTONE-CONTEXT.md`
(since GSD treats this file as transient per-milestone).

## Step 1 — Resolve the milestone

- If `$ARGUMENTS` is provided, use it as the milestone name.
- Else, read `.planning/MILESTONE-CONTEXT.md` and pull its name.
- Else, look at ROADMAP's `## Milestones` section for the `🚧` (in-progress)
  bullet — there should be exactly one.
- If none can be resolved, stop and ask the user which milestone.

## Step 2 — Compute which phases belong to this milestone

Open ROADMAP.md. The Phases section is organized by milestone — find the
section header for this milestone (e.g. `### 🚧 v1.1 Sharing (In Progress)`)
and collect every `### Phase N: ...` block that follows it, up to the next
milestone heading (or end of `## Phases`).

Call this set `milestonePhases`.

## Step 3 — Verify completion

For every phase in `milestonePhases`:

- Use `lib/roadmap.listPhases` to get its plans.
- Every plan must be `[x]` (done).
- For every plan, the corresponding SUMMARY file
  `.planning/phases/{phase-dir}/{phase}-{plan}-SUMMARY.md` must exist (GSD
  promise). If any is missing, list them and STOP — the user must run
  `/cp-execute-phase` or write SUMMARYs first.

Surface the verification as a checklist:
```
Verifying v1.1 Sharing:
  ✓ Phase 3 (Sharing):           2/2 plans done, 2/2 SUMMARYs present
  ✓ Phase 3.1 (Token expiry):    1/1 plans done, 1/1 SUMMARYs present
  ✓ Phase 4 (Export):            1/1 plans done, 1/1 SUMMARYs present
```

Only proceed if everything is `✓`.

## Step 4 — Roll up SUMMARYs into a milestone digest

Read every SUMMARY file in `milestonePhases`. Use `lib/frontmatter.parse` to
extract the GSD-shape frontmatter fields:

- `subsystem`
- `requires` / `provides` / `affects`
- `tech-stack.added`, `tech-stack.patterns`
- `key-files.created`, `key-files.modified`
- `key-decisions`
- `patterns-established`
- `requirements-completed`
- `duration`

Aggregate by unique-union (sets), then produce a milestone digest. Example
shape (this is what gets appended to `MILESTONES.md`):

```markdown
## {Milestone name}  — shipped {today's ISO date}

**Phases:** {first}-{last}    **Plans:** {count}    **Duration:** {sum or "—"}

**Requirements delivered:** REQ-04, REQ-05

**Subsystems touched:** sharing, export

**Tech added:** crypto.randomUUID, node-cron

**Key decisions:**
- {decision 1, source: phase 3 plan 01}
- {decision 2, source: phase 3.1 plan 01}

**Patterns established:**
- tokens TTL = 7d default
- cron jobs run every 5 min

**Files (created):** src/models/share.js, src/jobs/purge-tokens.js, src/api/export.js
**Files (modified):** prisma/schema.prisma, src/routes/index.js

**Phase summaries:**
- Phase 3: Sharing — see `.planning/phases/03-sharing/`
- Phase 3.1: Token expiry hotfix — see `.planning/phases/3.1-token-expiry-hotfix/`
- Phase 4: Export — see `.planning/phases/04-export/`
```

Show this digest to the user, ask for sign-off, then APPEND it (do not
overwrite) to `.planning/MILESTONES.md`. Append after the existing content,
preceded by a blank line.

## Step 5 — Collapse the milestone in ROADMAP.md

For the milestone's phase block(s) inside `## Phases`:

1. Replace the milestone's section header (e.g.
   `### 🚧 v1.1 Sharing (In Progress)`) plus all of its `### Phase N: ...`
   blocks with a single `<details>` wrapper:

   ```
   <details>
   <summary>✅ {Milestone name} (Phases {first}-{last}) — SHIPPED {today's ISO date}</summary>

   ### Phase N: {original heading and body verbatim}
   ...all phases...

   </details>
   ```

   Preserve the inner content byte-for-byte (GSD-compat: a switch back to
   GSD will still parse the same phase headings inside the `<details>`).

2. Update the `## Milestones` bullet for this milestone from `🚧 ` /
   `📋 ` to `✅ **{Milestone name}** — Phases {first}-{last} (shipped
   {today})`.

3. Rebuild the Progress table with `lib/roadmap.rebuildProgressTable`, which
   will mark each phase as Complete (since their plans are all `[x]`).

## Step 6 — Clean up transient milestone files

- DELETE `.planning/MILESTONE-CONTEXT.md` if it exists (GSD treats this file
  as belonging to the active milestone only — it's regenerated by
  `/cp-new-milestone`).
- DO NOT delete anything under `.planning/phases/` — those stay forever as
  the project history.

## Step 7 — Update STATE.md

- `## Current Position` →
  - `Phase: —` and `Plan: —` (or set to the first phase of the NEXT
    milestone if one is already listed in ROADMAP as `📋 `)
  - `Status: Milestone {name} shipped — awaiting next milestone`
  - `Last activity: {today} — shipped milestone {name}`
- `Progress:` → recompute against the whole project (donePlans / totalPlans)
- `Current focus:` (under Project Reference) → next milestone name or
  `(awaiting next milestone)`

Use `lib/state.updatePosition` and `lib/state.updateProgressBar`.

## Step 8 — Commit and report

If `cp.behavior.atomic_commits` is true, commit as:

```
cp: ship milestone {name} (phases X-Y)
```

Print:

```
✓ Milestone "{name}" shipped.
  Phases:        X-Y ({plan count} plans)
  Archived to:   .planning/MILESTONES.md
  Roadmap:       collapsed inside <details>
  Cleared:       .planning/MILESTONE-CONTEXT.md
  Project:       {donePlans}/{totalPlans} plans complete   {progressBar}

Next:
  /cp-new-milestone "{suggested next}"   # start the next milestone
  /cp-progress                            # confirm the new "you are here"
```

## Notes

- **GSD-compat hard contract:** do NOT change phase numbering, do NOT remove
  any phase block, do NOT alter frontmatter inside SUMMARY/PLAN files.
  Everything stays parseable by GSD's `gsd-complete-milestone`.
- This command is the only one in cp that DELETES a file (`MILESTONE-CONTEXT.md`).
  Show the user every change before writing.
- If the user wants to undo, the atomic commit makes `git revert HEAD` a
  one-shot rollback.
- Order matters: append to MILESTONES.md BEFORE editing ROADMAP.md, so a
  failure mid-flight leaves the archive intact and the roadmap unchanged.
